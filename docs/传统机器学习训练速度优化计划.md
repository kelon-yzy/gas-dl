## 1. 背景与现状

当前传统 ML 入口由 `src/pipeline/train_traditional.py` 调用 `src/ml/scripts/run_four_component_model_grid.py`，默认执行：

- 2 个 profile：`v3_raw_no_env`、`v3_raw_tph`
- 9 个组合：`svr_*`、`pls_*`、`xgboost_*`
- 3 个 seed：`42 / 52 / 62`
- 默认 `stacking_folds=5`
- 默认 `n_perturbations=24`
- 实验脚本外层 `--max-workers 1`

- train：`31696`

已有日志显示，`seed=42` 的 `svr_ridge` 与 `svr_pls` 单个 meta 组合各约 50 分钟，`svr_xgboost` 后续仍在运行中：

## 2. 瓶颈判断

### 2.1 RBF SVR 是第一瓶颈

当前位置：

- `src/ml/patent_model/modeling.py`
- `_make_pipeline()`
- `SVR(kernel="rbf", C=config.C, epsilon=config.epsilon, gamma=config.gamma)`

判断：

- scikit-learn 官方文档说明 `SVR` 拟合复杂度随样本数超过二次增长，几万级样本不适合作为主网格模型。
- 本项目 train 样本数约 `31696`，且每个组分、每个模态、每个 fold 都会重复拟合 SVR。

结论：

- `svr_*` 不适合作为全量正式网格默认项。
- `svr_*` 更适合作为小样本诊断、论文补充对照或最终少量精选配置。

### 2.2 OOF stacking 放大训练次数

当前位置：

- `src/ml/patent_model/modeling.py`
- `SingleComponentPatentModel._oof_meta_inputs()`
- `GroupKFold(n_splits=n_splits, shuffle=True, random_state=self.config.random_state)`

默认口径：

- `stacking_folds=5`
- 4 个组分
- 3 个模态分支
- 每个 fold 都要拟合分支模型，并且 fold 后还会全量拟合一次分支模型。

影响：

- 对 `PLS` 和 `XGBoost` 尚可接受。
- 对 `RBF SVR` 是乘法级放大。

### 2.3 动态权重 MC 扰动预测是第二瓶颈

当前位置：

- `src/ml/patent_model/modeling.py`
- `_compute_dynamic_weights()`

默认口径：

- `n_perturbations=24`
- 对每个 branch 构造 `(n_perturbations * n_samples, n_features)` 后批量预测。

正式测试集估算：

- 单组分单分支：`24 * 8304 = 199296` 条扰动预测
- 四组分三分支：约 `239 万` 条扰动预测
- 同一 branch 下 `ridge / pls / xgboost` 三个 meta 评估时会重复计算测试集动态权重

结论：

- 当前实现已经比逐次 predict 更好，但仍有重复计算空间。
- 测试集 `base_predictions + dynamic_weights` 应按 branch 配置缓存，供多个 meta learner 复用。

### 2.4 外层网格串行导致资源利用受限

当前位置：

- `experiments/exp01_traditional.ps1`
- `experiments/exp03_fusion_grid.ps1`
- `--max-workers 1`

判断：

- 内部 `fit_branch_stage` 和 fold 维度已有 joblib 线程并行。
- 外层是否并行需要根据 CPU 核数、内存、XGBoost 线程数共同决定。
- scikit-learn 文档提醒嵌套并行可能造成 oversubscription，因此不能简单把所有层都开满。

结论：

- 外层默认串行是保守但慢。
- 应提供明确的 CPU 配额策略，而不是固定 `--max-workers 1`。

### 2.5 XGBoost 线程默认保守

当前位置：

- `src/ml/scripts/train_patent_model.py`
- `--xgb-n-jobs` 默认 `1`
- `src/ml/patent_model/modeling.py`
- `XGBRegressor(..., n_jobs=config.xgb_n_jobs)`

判断：

- XGBoost 的 `n_jobs` 控制单模型并行线程数。
- 若外层串行，`xgb_n_jobs=1` 可能没有吃满 CPU。
- 若外层和 joblib 同时并行，`xgb_n_jobs` 过大会造成线程争用。

结论：

- 需要按执行模式设置 `xgb_n_jobs`。

## 3. 优化目标

### P0 目标

- 先让正式传统 ML 主网格可在可控时间内完成。
- 不牺牲主结果可比性。
- 保留 `svr_*` 的可复现对照路径，但不让它拖住全量实验。

### P1 目标

- 消除同一 branch 配置下不同 meta learner 的重复测试集动态权重计算。
- 给出训练耗时字段，后续结果表能直接比较速度。

### P2 目标

- 建立线程/进程资源配置规范。
- 对 full grid、smoke grid、paper grid 分别给出默认参数。

## 4. 分阶段实施计划

### Phase 0：记录基线耗时

任务：

- 在训练 summary 中增加耗时字段：`fit_seconds`、`evaluate_seconds`、`total_seconds`。
- 在 grid summary 中增加每个 combo 的耗时。
- 保留现有 `outputs/exp01_traditional/logs/train_seed42.log` 作为当前基线。

验收：

- 每个 `summary.json` 都包含耗时字段。
- `four_component_*_grid_summary.csv` 可以按 `total_seconds` 排序。

风险：

- 只加计时，不改训练逻辑，风险低。

### Phase 1：调整正式网格默认策略

任务：

- 将正式主网格拆成 `paper_core` 与 `svr_diagnostic` 两档。
- `paper_core` 默认包含：
  - `pls_ridge`
  - `pls_xgboost`
  - `xgboost_ridge`
  - `xgboost_xgboost`
- `svr_diagnostic` 默认包含：
  - `svr_ridge`
  - 可选 `svr_pls`
  - 可选 `svr_xgboost`
- `svr_diagnostic` 默认使用 `--train-limit` 或仅单 seed，避免拖住正式主线。

建议命令口径：

```powershell
python src\\pipeline\\train_traditional.py --data-dir outputs\\exp01_traditional --output-root outputs\\exp01_traditional --tag formal_seed42_core --seed 42 --profiles v3_raw_no_env v3_raw_tph --combo-list pls_ridge pls_xgboost xgboost_ridge xgboost_xgboost --max-workers 1 --n-jobs 2 --xgb-n-jobs 4
```

说明：

- 该参数透传现已完成，可以直接按 `combo-list` 调度 core 与 diagnostic。

验收：

- core grid 能独立运行。
- SVR 诊断可以单独运行，不影响 core grid 完成。

风险：

- 如果论文必须保留完整 `svr_*` 三 seed 全量结果，需要把 SVR 放入夜间/后台队列，而不是删除。

### Phase 2：缓存测试集动态权重

任务：

- 在同一 `ExecutionPlanItem` 内，`branch_artifacts` 已经被复用；进一步复用测试集预测产物。
- 针对同一 `branch_model_type + profile + seed + feature_profile`，缓存：
  - branch base predictions
  - dynamic weights
  - fixed average
  - dynamic average
- 让 `ridge / pls / xgboost` 三个 meta learner 只重复拟合 meta，不重复跑测试集 MC 扰动预测。

候选实现：

- 新增 `ComponentPredictionArtifacts` 或 `EvaluationBranchArtifacts`。
- 在 `MultiComponentPatentModel.evaluate()` 增加可选参数，例如 `prediction_cache`。
- 或拆出 `predict_branches()` 与 `predict_meta()` 两阶段 API。

验收：

- 同一 branch 的 3 个 meta 组合总评估时间明显下降。
- `component_metrics.csv`、`predictions.csv`、`dynamic_weights.csv` 输出格式不变。
- 随机扰动结果与旧实现数值一致或在可解释范围内一致。

风险：

- 需要小心不要把不同 meta 的最终预测缓存错。
- 缓存 key 必须包含 branch 模型、profile、seed、test split 与 `n_perturbations`。

### Phase 3：建立并行配置矩阵

任务：

- 增加一份推荐运行配置：
  - 单机小 CPU：`max_workers=1, n_jobs=-1, xgb_n_jobs=1`
  - 多核 CPU 跑 XGBoost 主网格：`max_workers=1-2, n_jobs=1-2, xgb_n_jobs=CPU/2`
  - 多 profile 并行：`max_workers=2, n_jobs=1, xgb_n_jobs=2-4`
- 在实验脚本中显式传入 `--n-jobs` 与 `--xgb-n-jobs`。
- 避免外层进程、joblib 线程、XGBoost 线程三层同时开满。

验收：

- `experiments/exp01_traditional.ps1` 不再隐藏依赖默认线程设置。
- 每次正式运行的资源参数记录到 `summary.json`。

风险：

- 机器差异会影响最佳参数，需要用 smoke run 校准。

### Phase 4：可选模型替代

任务：

- 若仍需要核方法对照，考虑添加：
  - `Nystroem + LinearSVR`
  - `RBFSampler + Ridge`
  - `LinearSVR`
  - `HistGradientBoostingRegressor`
- 保留 `SVR(RBF)` 作为小样本参考，不作为默认全量基线。

验收：

- 新模型有单独 combo 名称，不覆盖旧结果。
- 与旧 `svr_ridge` 在 smoke 数据上做误差和速度对照。

风险：

- 新模型会引入新的调参维度，优先级低于缓存和网格裁剪。

## 5. 推荐优先级

| 优先级 | 项目                                                                     | 预期收益 | 改动风险 |
| --- | ---------------------------------------------------------------------- | ---- | ---- |
| P0  | 加训练耗时字段                                                                | 中    | 低    |
| P0  | 正式网格排除全量 `svr_*`                                                       | 高    | 低    |
| P1  | `train_traditional.py` 透传 `--combo-list` / `--n-jobs` / `--xgb-n-jobs` | 中    | 低    |
| P1  | 测试集动态权重缓存                                                              | 高    | 中    |
| P2  | 并行配置矩阵和实验脚本参数化                                                         | 中    | 中    |
| P3  | 替代 RBF SVR 的近似核/线性模型                                                   | 中    | 中    |

## 8. 决策建议

建议先执行：

1. 增加耗时字段。
2. 让 `train_traditional.py` 透传 `--combo-list`、`--n-jobs`、`--xgb-n-jobs`。
3. 将正式主网格切到 `paper_core`。
4. 单独安排 `svr_diagnostic`。
5. 再做测试集动态权重缓存。

这个顺序的好处是：前 3 步能很快降低训练阻塞风险，同时不会碰核心建模逻辑；第 5 步收益高，但需要更仔细验证数值一致性，适合在主线先跑通后再做。
